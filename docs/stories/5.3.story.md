## Story

**As a** usuario que gestiona su semana,
**I want** ver en la Vista Semana mis actividades reales de los 7 días con su Time Budget semanal acumulado,
**so that** pueda tener visibilidad de mi carga de la semana completa sin tener que revisar día por día.

## Acceptance Criteria

### AC1 — Query `getActivitiesForWeek`

**Given** un usuario autenticado y una fecha cualquiera de la semana,
**When** se llama `getActivitiesForWeek(userId, date)`,
**Then:**
- Devuelve todas las `steps_activities` del usuario donde `scheduledAt` cae dentro de la semana ISO (Lunes 00:00:00Z – Domingo 23:59:59.999Z) que contiene `date`
- Usa `getWeekRange(date)` de `calendar-utils.ts` para calcular los límites (reutiliza lógica ya probada)
- Incluye `areaName` (LEFT JOIN `areas`) y `habitTitle` (LEFT JOIN `habits`) — `null` si no aplica
- Incluye `areaColor` derivado de `areas.maslowLevel` usando `maslowLevelToColor()` (reutiliza de `calendar.ts`)
- Ordena por `scheduledAt ASC`
- Está ubicada en `lib/db/queries/calendar.ts` (junto a `getActivitiesForDay`)
- Comparte la misma interfaz `ActivityForCalendar` — sin nuevo tipo

### AC2 — `page.tsx` carga actividades de la semana

**Given** el calendario carga la Vista Semana (vista por defecto),
**When** el Server Component renderiza,
**Then:**
- `app/(app)/calendar/page.tsx` llama `getActivitiesForWeek(user.id, new Date())` para la semana actual
- Convierte el resultado a `ICalendarEvent[]` usando el mismo mapping de Story 5.2 (inline, sin nueva función)
- Reemplaza la llamada a `getActivitiesForDay` — el Server Component ahora provee los 7 días completos al `CalendarClient`
- Si la query falla, renderiza el calendario con `events=[]` sin romper la página (error silencioso logueado)
- La Vista Día sigue funcionando correctamente con los eventos de la semana (ya filtra por día internamente)

### AC3 — Time Budget Semanal en Vista Semana

**Given** el usuario está en la Vista Semana,
**When** hay actividades programadas para esa semana,
**Then:**
- Bajo el toolbar del calendario aparece un panel "Time Budget Semanal" con:
  - **Comprometido:** suma de `scheduledDurationMinutes` de todas las actividades de la semana, en formato `Xh Ym`
  - **Disponible:** `112h 0m` (jornada activa estándar: 16h × 7 días = 6720 min)
  - **Libre:** diferencia entre disponible y comprometido (puede ser negativa → mostrar en rojo)
- El panel solo se muestra en Vista Semana (`view === 'week'`)
- Si no hay actividades, muestra `0h 0m` comprometido y `112h 0m` libre
- Reutiliza `calcTimeBudget()` y `formatMinutes()` — no duplicar lógica

### AC4 — WeekView muestra datos reales

**Given** el calendario muestra la Vista Semana,
**When** hay actividades en `steps_activities` para esa semana,
**Then:**
- Los eventos aparecen en las columnas correctas según su `scheduledAt` local (timezone local del navegador)
- Cada evento muestra su color de área (derivado de `maslowLevel`)
- El comportamiento es idéntico al ya implementado en WeekView — no hay regresión
- Al navegar a semanas anteriores/siguientes, el `CalendarClient` muestra los eventos del `events` array ya provisto (sin nueva query — se re-fetch solo si el usuario recarga la página)

### AC5 — Tests: query semanal + Time Budget semanal

**Given** la implementación completa,
**When** se ejecuta el suite de tests,
**Then:**
- `tests/unit/calendar/calendar-queries.test.ts` (archivo existente) agrega tests para `getActivitiesForWeek` con al menos 4 casos:
  1. Retorna actividades dentro de la semana ISO (lunes 00:00Z – domingo 23:59:59.999Z)
  2. Excluye actividades fuera de la semana (día antes del lunes, día después del domingo)
  3. Verifica límite de inicio de semana: lunes 00:00:00Z incluido
  4. Verifica límite de fin de semana: domingo 23:59:59.999Z incluido
- `tests/unit/calendar/time-budget.test.ts` (archivo existente) agrega tests para el cálculo semanal con al menos 2 casos:
  1. Cálculo correcto con `AVAILABLE_WEEKLY_MINUTES = 6720` (16h × 7)
  2. Libre negativo cuando comprometido semanal excede disponible
- Todos los tests anteriores pasan sin regresión (322+ → 330+ tests)

### Patrón de query (referencia: `getActivitiesForDay` en `lib/db/queries/calendar.ts`)

```typescript
// lib/db/queries/calendar.ts — agregar debajo de getActivitiesForDay
export async function getActivitiesForWeek(
  userId: string,
  date: Date
): Promise<ActivityForCalendar[]> {
  assertDatabaseUrl()

  const { start: weekStart, end: weekEnd } = getWeekRange(date)
  // getWeekRange retorna startOfWeek/endOfWeek de date-fns (ISO: lunes)
  // Necesitamos UTC boundaries para el query:
  const rangeStart = new Date(weekStart)
  rangeStart.setUTCHours(0, 0, 0, 0)
  const rangeEnd = new Date(weekEnd)
  rangeEnd.setUTCHours(23, 59, 59, 999)

  // Query idéntica a getActivitiesForDay — solo cambian los límites
  // (reutilizar el mismo patrón de SELECT + LEFT JOINs + WHERE + ORDER BY)
}
```

/** Mimics the week-range filter applied in getActivitiesForWeek */
function isInWeek(scheduledAt: Date, date: Date): boolean {
  const { start: weekStart, end: weekEnd } = getWeekRange(date)
  const rangeStart = new Date(weekStart); rangeStart.setUTCHours(0, 0, 0, 0)
  const rangeEnd = new Date(weekEnd); rangeEnd.setUTCHours(23, 59, 59, 999)
  return scheduledAt >= rangeStart && scheduledAt <= rangeEnd
}
