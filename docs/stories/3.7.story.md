## Story

**As a** usuario con KRs trimestrales definidos,
**I want** que el progreso de mis Key Results se calcule automáticamente desde mis activities completadas y time entries registradas,
**so that** no tenga que ingresar manualmente el avance de cada KR — el sistema mide mi ejecución real y actualiza el porcentaje de forma autónoma.

## Scope

**IN scope:**

- Nueva columna `okr_id` en tabla `steps_activities` (FK opcional a `okrs.id` — vincula una activity a un KR específico):
  - Migration en Supabase: `ALTER TABLE steps_activities ADD COLUMN okr_id UUID REFERENCES okrs(id) ON DELETE SET NULL`
  - Actualizar schema Drizzle en `lib/db/schema/steps-activities.ts`
- Query layer en `lib/db/queries/okrs.ts`:
  - `calculateKRProgress(userId, krId)` → calcula progreso según `kr_type`:
    - `time_based`: `SUM(time_entries.duration_seconds)` de activities vinculadas al KR → convierte a horas → `Math.min(100, Math.round((horasReales / targetValue) * 100))`
    - `outcome_based`: suma de activities completadas vinculadas → `Math.min(100, Math.round((completadas / targetValue) * 100))`
    - `milestone`: 100 si existe al menos 1 activity completada vinculada, 0 si no
  - `updateKRProgress(userId, krId)` → calcula y persiste `progress` en tabla `okrs`
  - `updateAnnualOKRProgress(userId, annualOkrId)` → promedio de `progress` de todos sus KRs activos
- Server Actions en `actions/okrs.ts`:
  - `recalculateKRProgress(krId)` — recalcula y guarda progreso de un KR específico
  - `confirmMilestone(krId)` — para KRs `milestone`: marca como completado creando una activity de confirmación vinculada
- UI en `components/okrs/AnnualOKRCard.tsx` (extensión de Story 3.6):
  - Barra de progreso de KR ya existe — ahora muestra datos reales (no 0%)
  - Botón "Confirmar hito" visible solo para KRs `milestone` con `progress === 0`
  - Barra de progreso del OKR anual actualizada con promedio real de KRs
- Recálculo automático: cuando se completa una activity, se recalcula el progreso del KR vinculado (via revalidatePath en Server Action existente o nuevo hook)
- Tests unitarios en `tests/unit/okrs/kr-progress.test.ts`:
  - `calculateKRProgress` para los 3 tipos de KR
  - Edge cases: 0 activities, progreso > 100% (cap), milestone sin confirmar

**OUT of scope:**
- Time tracking UI (Stories E5) — la Story 3.7 solo lee `time_entries` existentes
- Vinculación automática de activities a KRs por IA (futuro)
- Progreso en tiempo real vía Supabase Realtime (Story 5.9)
- Dashboard de progreso trimestral avanzado (Story 8.2)
- Edición masiva de vinculación activity ↔ KR

## Acceptance Criteria

1. **time_based:** Un KR `time_based` con `target_value=50` (horas) muestra `progress=40` si las activities vinculadas suman 20 horas en `time_entries`. [Source: FR4, lib/db/schema/time-entries.ts#durationSeconds] ✅
2. **outcome_based:** Un KR `outcome_based` con `target_value=10` muestra `progress=30` si hay 3 activities completadas vinculadas. [Source: FR4, lib/db/schema/okrs.ts#targetValue] ✅
3. **milestone:** Un KR `milestone` muestra `progress=0` sin confirmación y `progress=100` al presionar "Confirmar hito". [Source: FR4] ✅
4. **Progreso anual:** El progreso del OKR anual se actualiza automáticamente como promedio de sus KRs activos. [Source: lib/db/schema/okrs.ts#progress] ✅
5. **Cap 100%:** El progreso nunca supera 100%, incluso si las horas/items superan el `target_value`. [Source: FR4] ✅
6. **Sin ingreso manual:** El usuario no necesita escribir ningún número — el sistema calcula automáticamente desde `time_entries` y `steps_activities`. [Source: FR4] ✅
7. **Tests:** 100% de éxito en tests unitarios de lógica de cálculo de progreso para los 3 tipos de KR. ✅

## Dev Notes

### Insights Técnicos y Referencias

- **Schema `time_entries`:** `duration_seconds` INTEGER + `step_activity_id` FK. Total horas = `SUM(duration_seconds) / 3600`. [Source: lib/db/schema/time-entries.ts]
- **Schema `steps_activities`:** Añadir `okr_id UUID REFERENCES okrs(id) ON DELETE SET NULL`. La vinculación es opcional — una activity puede o no estar vinculada a un KR. [Source: lib/db/schema/steps-activities.ts]
- **Schema `okrs`:** Campo `progress INTEGER` ya existe (default 0). El recálculo hace UPDATE sobre este campo. [Source: lib/db/schema/okrs.ts#progress]
- **Ubicación:** Queries en `lib/db/queries/okrs.ts`, actions en `actions/okrs.ts`, tests en `tests/unit/okrs/`. [Source: docs/architecture/source-tree.md]
- **Recálculo en cascada:** `activity completada → recalculateKRProgress(kr.id) → updateAnnualOKRProgress(annual.id) → revalidatePath('/okrs')`
- **Sin Realtime en MVP:** El recálculo ocurre en el Server Action — no hay suscripción en tiempo real para el progreso (eso es Story 5.9).

### Lógica de Cálculo por Tipo

```typescript
// time_based: horas reales / horas objetivo
const totalSeconds = await sumTimeEntries(userId, krId)
const horasReales = totalSeconds / 3600
const progress = Math.min(100, Math.round((horasReales / kr.targetValue!) * 100))

// outcome_based: items completados / objetivo numérico
const completedCount = await countCompletedActivities(userId, krId)
const progress = Math.min(100, Math.round((completedCount / kr.targetValue!) * 100))

// milestone: binario — 0 o 100
const hasConfirmation = await hasCompletedActivity(userId, krId)
const progress = hasConfirmation ? 100 : 0
```

### UI Pattern

- El componente `AnnualOKRCard` ya renderiza `{kr.progress}%` y la barra de progreso — con los datos reales calculados, esto funciona automáticamente.
- El botón "Confirmar hito" solo aparece si `kr.krType === 'milestone' && kr.progress === 0`.
- Usar `useTransition` para `confirmMilestone` (ya establecido en el componente).

### Issues — MEDIUM (no bloqueantes para MVP)

**ISSUE-1: Migration Drift — okr_id ausente en archivos Drizzle locales** (MEDIUM)
La columna `okr_id` fue aplicada directamente en Supabase via MCP pero no fue capturada en archivos de migración Drizzle locales (`lib/db/migrations/`). El entorno de producción funciona, pero un CI con BD limpia o nuevo entorno fallaría.
**Acción requerida (antes de Story 8.2):** Ejecutar `drizzle-kit generate` para generar el migration file `.sql` correspondiente o crear manualmente `0001_add_okr_id_steps_activities.sql`.

**ISSUE-2: Recálculo automático incompleto** (MEDIUM)
El scope de la story menciona recálculo automático al completar una activity, pero `recalculateKRProgress` no está conectado a ningún callsite de completar activities ordinarias (solo funciona via `confirmMilestone`). El Server Action existe y está correcto; falta el punto de integración con el flujo de activities.
**Acción requerida (al implementar UI de activities, Epics E1/E5):** Invocar `recalculateKRProgress(kr.id)` al marcar una activity vinculada a un KR como `completed`.

### Concerns — LOW (no bloqueantes)

- **CONCERN-1:** `alert()` y `confirm()` nativos del browser en `AnnualOKRCard.tsx` — reemplazar con Toast/Dialog del design system en historia futura.
