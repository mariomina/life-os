## Story

**As a** usuario cuyo Inbox tiene items con IA no disponible (status='manual') o acumulados sin procesar >7 días,
**I want** poder clasificar y calendarizar items manualmente y ver una alerta cuando mi inbox se acumula,
**so that** nunca quede bloqueado por dependencia de IA y mantenga mi inbox bajo control.

## Acceptance Criteria

### AC1 — Función pura `getInboxAccumulationAlert`
**Given** una lista de InboxItems del usuario,
**When** alguno tiene `status='pending'` o `status='manual'` y `createdAt < (now - 7 días)`,
**Then** la función retorna `{ type: 'warning', count: N, oldestDays: D }`,
Y si ninguno cumple la condición, retorna `null`.

### AC2 — UI: Banner de alerta en Inbox
**Given** que `getInboxAccumulationAlert` retorna un resultado no-null,
**When** se renderiza `InboxClient`,
**Then** se muestra un banner amarillo/naranja al tope de la lista:
"Tienes N items sin procesar desde hace D días. Revísalos."
Con botón "Ver pendientes" que activa el filtro `'pending'`.

### AC3 — Server Action `updateInboxItemManually`
**Given** un inbox item con cualquier status (típicamente `'manual'` o `'pending'`),
**When** el usuario llama a `updateInboxItemManually(itemId, data)`,
**Then** se actualizan los campos del item con los datos manuales:
- `aiClassification = data.classification`
- `aiSuggestedAreaId = data.areaId`
- `aiSuggestedTitle = data.title ?? rawText`
- `aiSuggestedSlot = data.slot ?? null`
- `aiSuggestedDurationMinutes = data.durationMinutes ?? null`
- `status = 'processed'`
- `processedAt = now()`
Y retorna `{ success: true }`.

### AC4 — Ownership guard en `updateInboxItemManually`
**Given** cualquier llamada a `updateInboxItemManually`,
**When** el item no existe o pertenece a otro usuario,
**Then** retorna `{ success: false, error: 'Item no encontrado' }`.

### AC5 — Validación de campos en `updateInboxItemManually`
**Given** una llamada a `updateInboxItemManually` sin `classification` o sin `areaId`,
**When** se ejecuta la acción,
**Then** retorna `{ success: false, error: '...' }` con mensaje descriptivo.

### AC6 — UI: `ManualProcessingCard` para items `status='manual'`
**Given** un inbox item con `status='manual'`,
**When** se renderiza la lista del Inbox,
**Then** se muestra una tarjeta con:
- Selector "Tipo" (classification: task/event/project/idea/reference/habit)
- Selector "Área" (dropdown con áreas del usuario)
- Input "Fecha/hora" (datetime-local para slot)
- Input "Duración" (número en minutos, opcional)
- Botón **"✓ Guardar"** que llama a `updateInboxItemManually`
- Mensaje de error IA si `aiError` existe (informativo)

### AC7 — Flujo post-manual: reutilización de ProposalCard
**Given** un item que pasa de `status='manual'` a `status='processed'` vía `updateInboxItemManually`,
**When** la página se revalida,
**Then** el item muestra `ProposalCard` o `ProjectProposalCard` (si `classification='project'`)
para que el usuario confirme con 1-click (reutilizando flujo de Stories 6.3 y 6.4).

### AC8 — Areas cargadas en InboxPage
**Given** que `ManualProcessingCard` necesita la lista de áreas del usuario,
**When** se carga `InboxPage` (Server Component),
**Then** se llama a `getUserAreas(userId)` en paralelo con `getInboxItemsByUser` y `getSystemWorkflowTemplates`,
Y `InboxClient` recibe la prop `areas`.

### AC9 — Tests ≥ 495 + nuevos de Story 6.5
**Given** la implementación completa,
**When** se ejecuta `npm test`,
**Then** todos los tests pasan (≥ 495 anteriores + nuevos tests de `getInboxAccumulationAlert` y `updateInboxItemManually`).

## Risks

- **Areas vacías:** Si el usuario no tiene áreas configuradas, el dropdown estará vacío. Mostrar mensaje "Configura tus áreas primero" en ese caso.
- **`status='manual'` ya con `aiSuggestedAreaId`:** No debería ocurrir (la IA falló antes de poner datos), pero `updateInboxItemManually` sobreescribe sin problema.
- **Slot opcional:** El usuario puede no querer calendarizar (ej: idea o referencia). `aiSuggestedSlot` es nullable — aceptar sin slot.

## Dev Notes

### Función pura `getInboxAccumulationAlert`

Archivo nuevo: `features/inbox/alerts.ts`

```typescript
import type { InboxItem } from '@/lib/db/schema/inbox-items'

export interface InboxAccumulationAlert {
  type: 'warning'
  count: number    // número de items >7 días sin cerrar
  oldestDays: number   // días desde el item más antiguo
}

const ACCUMULATION_THRESHOLD_DAYS = 7

export function getInboxAccumulationAlert(
  items: InboxItem[]
): InboxAccumulationAlert | null {
  const now = Date.now()
  const threshold = ACCUMULATION_THRESHOLD_DAYS * 24 * 60 * 60 * 1000

  const oldItems = items.filter((item) => {
    if (item.status !== 'pending' && item.status !== 'manual') return false
    return now - new Date(item.createdAt).getTime() > threshold
  })

  if (oldItems.length === 0) return null

  const oldestMs = Math.max(...oldItems.map((i) => now - new Date(i.createdAt).getTime()))
  const oldestDays = Math.floor(oldestMs / (24 * 60 * 60 * 1000))

  return { type: 'warning', count: oldItems.length, oldestDays }
}
```

### Server Action `updateInboxItemManually`

Archivo: `actions/inbox.ts` (añadir al final)

```typescript
export interface ManualUpdateData {
  classification: 'task' | 'event' | 'project' | 'habit' | 'idea' | 'reference'
  areaId: string
  title?: string
  slot?: string | null  // ISO 8601 o null
  durationMinutes?: number | null
}

export interface ManualUpdateResult {
  success: boolean
  error?: string
}

export async function updateInboxItemManually(
  itemId: string,
  data: ManualUpdateData
): Promise<ManualUpdateResult>
```

Pasos:
1. Validar `itemId` + `data.classification` + `data.areaId`
2. `assertDatabaseUrl()` + `getAuthenticatedUserId()`
3. Query con ownership: `WHERE id=itemId AND userId=userId LIMIT 1`
4. Guard: item not found → error
5. Update: `aiClassification`, `aiSuggestedAreaId`, `aiSuggestedTitle ?? rawText`, `aiSuggestedSlot`, `aiSuggestedDurationMinutes`, `status='processed'`, `processedAt=now()`, `updatedAt=now()`
6. `revalidatePath('/inbox')`
7. Return `{ success: true }`

**Nota:** `confirmInboxProposal` y `createProjectFromInbox` funcionan sobre items `status='processed'`, por lo que después de `updateInboxItemManually`, el usuario puede usar el flujo normal de confirmación.
