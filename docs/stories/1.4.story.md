## Story

**As a** data engineer working on life-os,
**I want** the Drizzle schema refined with missing performance indexes, additional DB-level constraints, proper FK enforcement to `auth.users`, seed data for system workflow templates, and production-ready migrations generated via `drizzle-kit generate`,
**so that** the database is performant, consistent, and pre-populated with the 8 MVP system templates that the application requires to function correctly for all users.

## Dependencies

- **Requires:** Story 1.2 — Status: Done ✅ (Proyecto Supabase existente con 15 tablas creadas via `drizzle-kit push` y RLS habilitado en todas las tablas)
- **Blocks:** Story 1.5 (onboarding flow necesita seed de `areas` por usuario — depende de schema estable), Story 1.6 (tests contra DB necesitan schema finalizado)

## Scope

**IN scope:**
- Revisión y análisis del schema Drizzle existente en `lib/db/schema/` (15 archivos)
- Adición de índices de performance faltantes en columnas de alta consulta:
  - `areas`: índice en `(user_id)` y `(user_id, maslow_level)`
  - `okrs`: índice en `(user_id, status)`, `(user_id, type)`, `(parent_id)`
  - `habits`: índice en `(user_id, is_active)`, `(user_id, area_id)`
  - `projects`: índice en `(user_id, status)`, `(user_id, area_id)`
  - `workflows`: índice en `(project_id)`, `(user_id, status)`
  - `tasks`: índice en `(workflow_id, order)`, `(user_id, status)`
  - `correlations`: ya tiene índice en `(user_id, is_active, computed_at)` — verificar suficiencia
- FK explícita `user_id → auth.users(id)` via SQL migration (Drizzle no puede referenciar `auth.users` natively)
- Constraint `CHECK` para validar rangos de valores críticos (maslow_level 1-8, progress 0-100, energy_level 1-5)
- Generación de migration files via `drizzle-kit generate` para flujo producción
- Seed data para `workflow_templates` con los 8 templates MVP del sistema (`is_system=true`)
- Script SQL documentado para seeds: `supabase/seeds/workflow-templates.sql`
- Documentación del schema final en `docs/architecture/db-schema.md`
- Verificación de `npm run typecheck` y `npm run build` sin errores post-cambios

**OUT of scope:**
- Cambios en la lógica de RLS (ya implementada en Story 1.2 — `supabase/rls-policies.sql`)
- Nuevas tablas (schema actual de 15 tablas es completo para MVP)
- Trigger functions / stored procedures (futura story del motor de correlaciones)
- Supabase Realtime configuration (story separada)
- Testing framework (Story 1.6)
- Seed data de áreas por usuario (se hace en onboarding flow — Story 1.5)
- Edge Functions para cron jobs (story separada del motor de correlaciones)

## Complexity

**Estimación:** S — 3 story points
- Razón: Cambios son aditivos (nuevos índices, constraints, seeds) sobre schema estable — sin breaking changes. La parte más compleja es el seed de `workflow_templates` (JSONB con `tasks_config` anidado). FK a `auth.users` requiere SQL raw fuera de Drizzle. Todo el trabajo es en `lib/db/schema/` y SQL scripts.

## Risks

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| `drizzle-kit generate` detecta drift con `push` previo | Media | Medio | Ejecutar `drizzle-kit push --force` en dev primero para sincronizar; luego `generate` para producción |

| CHECK constraint rompe datos existentes en dev | Baja | Bajo | DB está vacía en dev (sin datos de usuario reales) — constraint aplica sin conflicto |

### FK a auth.users — implementación SQL

[Source: Supabase docs + comentario en areas.ts]

Drizzle ORM no puede referenciar directamente `auth.users` porque está en un schema diferente. La FK debe aplicarse via SQL raw en una migration de Supabase:

```sql
-- supabase/migrations/add_auth_fk_and_constraints.sql
-- FK: user_id → auth.users(id) con CASCADE para limpiar datos si usuario es eliminado
ALTER TABLE public.areas
  ADD CONSTRAINT areas_user_id_fkey
  FOREIGN KEY (user_id) REFERENCES auth.users(id) ON DELETE CASCADE;

-- Repetir para cada tabla con user_id...
-- (areas, area_scores, okrs, habits, skills, workflow_templates,
--  projects, workflows, tasks, steps_activities, time_entries,
--  inbox_items, checkin_responses, step_skill_tags, correlations)
```

**IMPORTANTE:** `workflow_templates` tiene `user_id` nullable (null para templates del sistema). La FK debe ser:
```sql
ALTER TABLE public.workflow_templates
  ADD CONSTRAINT workflow_templates_user_id_fkey
  FOREIGN KEY (user_id) REFERENCES auth.users(id) ON DELETE CASCADE;
-- NULL user_id pasa la constraint (FK solo verifica NON-NULL values)
```

### drizzle-kit generate vs push — diferencias clave

[Source: docs/architecture/tech-stack.md + convenciones del proyecto]

| Comando | Uso | Produce |
|---------|-----|---------|
| `drizzle-kit push` | Dev — aplica cambios directamente (Story 1.2) | Sin archivos de migration |
| `drizzle-kit generate` | Producción — genera SQL migration file | `drizzle/migrations/*.sql` |
| `drizzle-kit migrate` | CI/CD — aplica migration files al DB | Aplica migration files |

**Para esta story:** usar `generate` + `migrate` para los cambios de índices en schema Drizzle. Los scripts SQL de FK/constraints y seeds se aplican vía Supabase MCP o SQL Editor directamente (son parte del `supabase/migrations/` y `supabase/seeds/` directories, no del flujo drizzle-kit).
