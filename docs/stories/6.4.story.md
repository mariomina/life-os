## Story

**As a** usuario con un item de Inbox clasificado como `project` por la IA,
**I want** ver una propuesta para convertirlo en un Proyecto con template de workflow,
**so that** pueda lanzar un proyecto estructurado directamente desde el Inbox con 1 clic.

## Acceptance Criteria

### AC1 — Migración: `project_id` en `inbox_items`
**Given** que `inbox_items` no tiene campo de vínculo a proyectos,
**When** se aplica la migración,
**Then** existe la columna `project_id uuid NULLABLE FK → projects(id) ON DELETE SET NULL` en la tabla `inbox_items`,
Y el schema Drizzle de `inbox-items.ts` incluye el campo `projectId`.

### AC2 — Server Action `createProjectFromInbox`
**Given** un inbox item con `status='processed'` y `aiClassification='project'`,
**When** el usuario llama a `createProjectFromInbox(itemId, templateId?)`,
**Then** se inserta un nuevo registro en `projects` con:
- `title = aiSuggestedTitle ?? rawText`
- `areaId = aiSuggestedAreaId`
- `okrId = aiSuggestedOkrId ?? null`
- `templateId = templateId ?? null`
- `status = 'active'`
Y `inbox_items.projectId` se actualiza con el nuevo ID,
Y retorna `{ success: true, projectId }`.

### AC3 — Ownership guard
**Given** cualquier llamada a `createProjectFromInbox`,
**When** el item no existe o pertenece a otro usuario,
**Then** retorna `{ success: false, error: 'Item no encontrado' }`.

### AC4 — Guard de estado
**Given** un item con `status` distinto de `'processed'`,
**When** se llama a `createProjectFromInbox`,
**Then** retorna `{ success: false, error: 'El item no tiene propuesta IA lista' }`.

### AC5 — Guard de doble conversión
**Given** un item que ya tiene `projectId` (ya fue convertido),
**When** se llama de nuevo a `createProjectFromInbox`,
**Then** retorna `{ success: false, error: 'Este item ya fue convertido en proyecto' }`.

### AC6 — Guard de área requerida
**Given** un item con `aiSuggestedAreaId = null`,
**When** se llama a `createProjectFromInbox`,
**Then** retorna `{ success: false, error: 'El item no tiene área sugerida' }`.

### AC7 — UI: tarjeta de proyecto emergente
**Given** un item con `status='processed'`, `aiClassification='project'` y sin `projectId`,
**When** la lista del Inbox se renderiza,
**Then** se muestra una `ProjectProposalCard` con:
- Título sugerido (`aiSuggestedTitle`)
- Área vinculada
- Selector de template (lista de templates del sistema o "Sin template")
- Botón **"✓ Crear Proyecto"** que llama a `createProjectFromInbox`
- Botón **"Descartar"**

### AC8 — UI: feedback visual tras creación
**Given** el usuario hace clic en "✓ Crear Proyecto",
**When** `createProjectFromInbox` retorna `{ success: true }`,
**Then** la tarjeta muestra "✓ Proyecto creado. Ver en Proyectos."
Y el item en la lista actualiza su badge visualmente.

### AC9 — Revalidación de rutas
**Given** una creación exitosa,
**When** `createProjectFromInbox` completa el insert,
**Then** se llama `revalidatePath('/inbox')` y `revalidatePath('/projects')`.

### AC10 — Tests ≥ 475 + nuevos de detección de proyecto
**Given** la implementación completa de Story 6.4,
**When** se ejecuta `npm test`,
**Then** todos los tests pasan (≥ 475 anteriores + nuevos tests de `createProjectFromInbox`).

## Business Value

Cierra el loop del Epic 6 para items de tipo `project`: el usuario captura "Lanzar curso de TypeScript" y en 1 clic tiene un proyecto estructurado en su sistema de productividad, listo para añadir tareas y steps.

## Risks

- **Templates vacíos en DB:** Si no hay templates sembrados (`workflow_templates` vacía), el selector mostraría "Sin templates". Mitigación: la seed ya debe existir o el campo es opcional (templateId nullable).
- **`aiClassification='project'` poco frecuente en MVP:** El LLM puede clasificar como `task` en lugar de `project` para textos ambiguos. Es una decisión de modelo, no de código.
- **`inbox_items` ya tiene `stepActivityId`:** El campo `projectId` es paralelo — ambos pueden ser null; un item no puede ser a la vez activity y proyecto (guardado por lógica de UI, no DB constraint en MVP).

## Dev Notes

### Migración necesaria

```sql
ALTER TABLE inbox_items
  ADD COLUMN IF NOT EXISTS project_id uuid
  REFERENCES projects(id) ON DELETE SET NULL;

CREATE INDEX IF NOT EXISTS idx_inbox_items_project
  ON inbox_items(project_id)
  WHERE project_id IS NOT NULL;
```

### Schema Drizzle — `inbox-items.ts`

Añadir import de `projects` y campo `projectId`:

```typescript
import { projects } from './projects'

// En el body de inboxItems:
projectId: uuid('project_id').references(() => projects.id, {
  onDelete: 'set null',
}),

// En los indexes:
projectIdx: index('inbox_items_project_idx').on(table.projectId),
```

### Server Action `createProjectFromInbox`

Archivo: `actions/inbox.ts` (añadir al final, junto a `confirmInboxProposal`)

```typescript
export interface CreateProjectFromInboxResult {
  success: boolean
  projectId?: string
  error?: string
}

export async function createProjectFromInbox(
  itemId: string,
  templateId?: string
): Promise<CreateProjectFromInboxResult>
```

Pasos internos:
1. `assertDatabaseUrl()` + `getAuthenticatedUserId()`
2. Query: `SELECT * FROM inbox_items WHERE id=itemId AND userId=userId LIMIT 1`
3. Guard: item not found → error
4. Guard: `item.status !== 'processed'` → error
5. Guard: `item.projectId != null` → error (doble conversión)
6. Guard: `!item.aiSuggestedAreaId` → error (área requerida)
7. Insert en `projects`:
   - `userId`, `areaId: item.aiSuggestedAreaId`
   - `title: item.aiSuggestedTitle ?? item.rawText`
   - `okrId: item.aiSuggestedOkrId ?? null`
   - `templateId: templateId ?? null`
   - `status: 'active'`
8. Update `inbox_items SET projectId=newProject.id, updatedAt=now() WHERE id=itemId`
9. `revalidatePath('/inbox')`, `revalidatePath('/projects')`
10. Return `{ success: true, projectId: newProject.id }`

### UI Changes en `InboxClient.tsx`

1. **Import**: `createProjectFromInbox` de `@/actions/inbox` + `getSystemWorkflowTemplates` (o se carga en la página Server Component y se pasa como prop)
2. **Estado adicional**: `selectedTemplateId: string | null` por item (o en estado local del `ProjectProposalCard`)
3. **Handler `handleCreateProject(itemId, templateId?)`**: llama a `createProjectFromInbox`, actualiza `confirmedProjectId` similar a `confirmedId`
4. **`ProjectProposalCard` sub-componente**:
   - Props: `item`, `templates`, `onCreateProject`, `onDiscard`, `isLoading`, `confirmedProjectId`
   - Dropdown de templates (select nativo con opciones)
   - Botón "✓ Crear Proyecto"

**Nota de carga de templates:**
Los templates del sistema son globales. Para evitar un `useEffect` fetch en el cliente, se pueden pasar como prop desde `InboxPage` (Server Component) usando `getSystemWorkflowTemplates()`.

Cambio en `InboxPage` (`app/(app)/inbox/page.tsx`):
```typescript
const [items, templates] = await Promise.all([
  getInboxItemsByUser(user.id),
  getSystemWorkflowTemplates(),
])
// Pasar templates a InboxClient
```

Cambio en `InboxClient`:
```typescript
interface InboxClientProps {
  initialItems: InboxItem[]
  templates: WorkflowTemplate[]  // NUEVO
}
```
