## Story

**As a** usuario con áreas y actividades registradas,
**I want** poder crear, ver, editar y archivar mis habilidades con nombre, nivel y área de vida vinculada,
**so that** pueda llevar un registro de las competencias que estoy desarrollando y verlas en un listado ordenado por nivel.

## Acceptance Criteria

### AC1 — Server Action `createSkill`
**Given** datos válidos (name + level + userId),
**When** el usuario llama a `createSkill({ name, level, areaId? })`,
**Then** se inserta en `skills` con `autoDetected=false` y retorna `{ success: true, skillId }`.

### AC2 — Guard: nombre duplicado en `createSkill`
**Given** el usuario ya tiene una habilidad con el mismo nombre,
**When** intenta crear otra con el mismo nombre,
**Then** retorna `{ success: false, error: 'Ya existe una habilidad con ese nombre' }`.

### AC3 — Server Action `updateSkill`
**Given** una habilidad existente del usuario,
**When** llama a `updateSkill(skillId, { name?, level?, areaId? })`,
**Then** actualiza los campos indicados y retorna `{ success: true }`.
Y si el skill no pertenece al usuario, retorna `{ success: false, error: 'Skill no encontrada' }`.

### AC4 — Server Action `archiveSkill`
**Given** una habilidad existente,
**When** el usuario llama a `archiveSkill(skillId)`,
**Then** el skill se marca como archivado (status='archived') y retorna `{ success: true }`.
(No borrado físico — preserva historial de tiempo invertido.)

### AC5 — Query `getUserSkills`
**Given** un usuario autenticado,
**When** se llama a `getUserSkills(userId)`,
**Then** retorna array de Skills activas del usuario ordenadas por `name asc`.

### AC6 — UI: página `/skills` con listado
**Given** el usuario navega a `/skills`,
**When** tiene skills registradas,
**Then** se muestra la lista con nombre, nivel (badge coloreado), área vinculada y tiempo invertido (`0h` si no hay time entries).
Y si no tiene skills, se muestra estado vacío "No has registrado habilidades aún."

### AC7 — UI: formulario de creación en `/skills`
**Given** el usuario está en `/skills`,
**When** hace clic en "Nueva habilidad",
**Then** aparece un formulario con:
- Input nombre (requerido)
- Select nivel: Beginner / Intermediate / Advanced / Expert (default Beginner)
- Select área (opcional, dropdown de áreas del usuario)
- Botón "Crear"

### AC8 — Tests ≥ 518 + nuevos de Story 7.1
**Given** la implementación completa,
**When** se ejecuta `npm test`,
**Then** todos los tests pasan (≥ 518 anteriores + nuevos tests de `createSkill`, `updateSkill`, `archiveSkill`).

## Risks

- **Schema `archived` no existe:** La tabla `skills` no tiene campo `status`. Necesita migración `ALTER TABLE skills ADD COLUMN status text DEFAULT 'active'`. O bien se usa `autoDetected` como indicador y el "archivado" se implementa con un campo nuevo. → Usar nueva columna `archivedAt timestamp` (nullable — null = activo).
- **Nombre único por usuario:** El schema ya tiene `uniqueIndex('skills_user_id_name_idx')` — Drizzle lanzará error de constraint en insert duplicado. Capturarlo y retornar error descriptivo.

**`createSkill`:**
1. Validar name (no vacío)
2. `assertDatabaseUrl()` + `getAuthenticatedUserId()`
3. Insert en `skills` — catch unique constraint → `{ success: false, error: 'Ya existe...' }`
4. Return `{ success: true, skillId: newSkill.id }`

**`updateSkill`:**
1. Ownership check (WHERE id + userId)
2. Update campos provistos
3. Return `{ success: true }` o `{ success: false, error: 'Skill no encontrada' }`

**`archiveSkill`:**
1. Ownership check
2. Update `archivedAt = now()`
3. Return `{ success: true }`

### Query `getUserSkills`

Archivo nuevo: `lib/db/queries/skills.ts`

```typescript
export async function getUserSkills(userId: string): Promise<Skill[]> {
  return db
    .select()
    .from(skills)
    .where(and(eq(skills.userId, userId), isNull(skills.archivedAt)))
    .orderBy(asc(skills.name))
}
```
