## Story

**As a** usuario activo con áreas de vida configuradas,
**I want** que el sistema me bloquee blandamente la creación de OKRs vinculados a áreas de nivel 7-8 (B-Needs avanzados) cuando algún área de nivel 1-2 (D-Needs críticos) tiene score <50% por más de 14 días consecutivos,
**so that** el sistema me recuerde priorizar mis necesidades básicas antes de perseguir objetivos de crecimiento y autotrascendencia.

## Scope

**IN scope:**
- Lógica pura de validación en `features/maslow/hierarchy-guard.ts`:
  - `checkHierarchyBlock(areas, areaScoreHistory)` → `HierarchyBlockResult`
  - Detección: área nivel 1-2 con `currentScore < 50` por >14 días consecutivos
  - Resultado: lista de áreas bloqueantes con score actual y días en crisis
- Hook de UI `useHierarchyGuard(areas)` → estado de bloqueo para formularios de OKR
- Componente `HierarchyWarning` — banner explicativo rojo con área(s) bloqueante(s)
- Integración del bloqueo en el formulario de creación de OKR (`type='annual'` vinculado a área nivel 7-8)
- Tests unitarios: toda la lógica de `hierarchy-guard.ts`
- TypeScript strict, sin `any`

**OUT of scope:**
- CRUD completo de OKRs (Stories 3.5-3.6)
- Bloqueo en KRs trimestrales (solo OKRs anuales vinculados a nivel 7-8)
- Persistencia del estado de bloqueo en DB (calculado en runtime desde `area_scores`)
- Desbloqueo manual por el usuario
- Configuración del umbral por el usuario (50% y 14 días son fijos en MVP)

## Complexity

**Estimación:** S — 2 story points
- Razón: Lógica pura similar a Story 3.2. Reutiliza `getUserAreas` y `getRecentAreaScores` ya existentes. La validación es cálculo sobre datos disponibles + UI de bloqueo condicional.

## Risks

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| `area_scores` sin historial suficiente (<14 días) | Alta | Bajo | Si no hay 14 días de historial → NO bloquear (beneficio de la duda) |
| `currentScore` desactualizado en tabla `areas` | Media | Bajo | Usar historial de `area_scores` como fuente de verdad para el cálculo de días |
| Falsos positivos por días discontinuos <50% | Media | Bajo | Requerir 14 días *consecutivos* — no acumulados |

## Acceptance Criteria

1. `checkHierarchyBlock(areas, scoreHistory)` retorna array de áreas bloqueantes correctamente tipado
2. Área nivel 1-2 con `currentScore < 50` por >14 días consecutivos → resultado contiene esa área como bloqueante
3. Área nivel 1-2 con `currentScore < 50` pero solo 10 días → NO bloquea (umbral no alcanzado)
4. Área nivel 1-2 con `currentScore >= 50` → NO bloquea aunque lleve >14 días
5. Área nivel 3-8 con score bajo → NO bloquea (solo niveles 1-2 activan el bloqueo)
6. Sin historial de 14 días → NO bloquea (beneficio de la duda)
7. `HierarchyWarning` muestra nombre del área bloqueante, score actual y días en crisis
8. Formulario de OKR anual vinculado a área nivel 7-8 muestra `HierarchyWarning` y deshabilita el botón "Crear" si hay bloqueo activo
9. Formulario de OKR anual vinculado a área nivel 1-6 NO se ve afectado por el bloqueo
10. `npm run typecheck` PASS, `npm run lint` PASS (0 errores)
11. Tests unitarios PASS: todos los casos de `checkHierarchyBlock`

export interface BlockedArea {
  areaId: string

export interface HierarchyBlockResult {
  isBlocked: boolean
  blockedAreas: BlockedArea[]
}

### Lógica de detección

```typescript
const CRISIS_SCORE_THRESHOLD = 50      // score < 50%
const CRISIS_DAYS_THRESHOLD = 14       // días consecutivos
const BLOCKING_LEVELS = new Set([1, 2]) // solo D-Needs críticos

// Algoritmo:
// 1. Filtrar áreas con maslowLevel 1-2
// 2. Para cada área: obtener historial de scores ordenado por fecha desc
// 3. Contar días consecutivos con score < 50 desde el más reciente
// 4. Si consecutiveDays >= 14 → área bloqueante
```

`getRecentAreaScores(userId, days=30)` ya existe en `lib/db/queries/area-scores.ts`.
Retorna `AreaScore[]` con `areaId`, `score`, `scoredAt`.
Agrupar por `areaId` → array de scores ordenados desc → contar racha consecutiva <50.
