## Story

**As a** usuario que quiere mantener accountability sobre su vida diaria,
**I want** ver al abrir la app un banner persistente con las actividades programadas del día anterior que aún no han sido confirmadas, poder marcarlas como completadas, omitidas o pospuestas con un solo tap, y confirmar en lote los hábitos recurrentes,
**so that** el sistema tenga datos reales de lo que hice cada día — actualizando automáticamente los streaks de hábitos, scores de área y el registro histórico — sin que yo tenga que recordar hacerlo manualmente.

- [x] Cada item tiene tres acciones: **✓ Completado**, **✗ Omitido**, **↷ Posponer**

### AC4 — Confirmar en lote hábitos (bulk confirm)

### Query `getUncheckedActivities` — lógica de deduplicación

```typescript
// Actividades del día anterior sin respuesta en checkin_responses:
const dayStart = new Date(date)
dayStart.setUTCHours(0, 0, 0, 0)
const dayEnd = new Date(date)
dayEnd.setUTCHours(23, 59, 59, 999)

const rows = await db
  .select({ activity: stepsActivities, areaName: areas.name, habitTitle: habits.title })
  .from(stepsActivities)
  .leftJoin(areas, eq(stepsActivities.areaId, areas.id))
  .leftJoin(habits, eq(stepsActivities.habitId, habits.id))
  .where(
    and(
      eq(stepsActivities.userId, userId),
      eq(stepsActivities.status, 'pending'),
      gte(stepsActivities.scheduledAt, dayStart),
      lte(stepsActivities.scheduledAt, dayEnd)
    )
  )
  .orderBy(
    // hábitos primero (habitId IS NOT NULL → false < true en ORDER BY DESC)
    desc(isNotNull(stepsActivities.habitId)),
    asc(stepsActivities.scheduledAt)
  )
  .limit(PAGE_SIZE) // 5 por defecto
```

**Nota:** La deduplicación contra `checkin_responses` se hace con un `notExists` subquery o filtrando en memoria. Preferir subquery para correctness en edge cases.

### confirmActivity — lógica completa

```typescript
export async function confirmActivity(
  activityId: string,
  status: 'completed' | 'skipped' | 'postponed'
): Promise<ActionResult> {
  // 1. Auth check
  // 2. Cargar stepsActivities row (verificar userId)
  // 3. Upsert checkin_responses (stepActivityId + checkinDate = yesterday)
  // 4. Update steps_activities.status
  //    completed → 'completed', completedAt = now()
  //    skipped   → 'skipped'
  //    postponed → status queda 'pending', scheduledAt = today (reagenda)
  // 5. Si habitId existe:
  //    - Cargar todas las completions del hábito (checkin_responses completed para ese habitId)
  //    - Llamar calculateStreak(completions, habit.rrule)
  //    - Update habits.streakCurrent, habits.streakBest
  //    - Si completed: update habits.lastCompletedAt = yesterday
  //    - Si skipped: streakCurrent = 0 (no update best)
  //    - Si postponed: no tocar streak
  // 6. revalidatePath('/')
}
```

### Optimistic UI en `DailyCheckinBanner`

```typescript
// Estado local: Map<activityId, 'completed' | 'skipped' | 'postponed' | null>
const [responses, setResponses] = useState<Map<string, string>>(() => new Map())

// Al presionar una acción:
function handleConfirm(activityId: string, status: string) {
  setResponses(prev => new Map(prev).set(activityId, status)) // optimistic
  startTransition(async () => {
    const result = await confirmActivity(activityId, status)
    if (result.error) {
      setResponses(prev => {
        const next = new Map(prev)
        next.delete(activityId) // revert
        return next
      })
      setError(result.error)
    }
  })
}
```
