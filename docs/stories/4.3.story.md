## Story

**As a** usuario que crea un nuevo proyecto o accede al canvas de workflow,
**I want** poder seleccionar uno de los 8 templates predefinidos de workflow (Personal Development, Product Launch, Health Sprint, etc.),
**so that** el canvas se instancie automáticamente con Tasks y Steps predefinidos que reflejan el template elegido, ahorrándome diseñar el workflow desde cero.

## Dependencies

- **Requires:** Story 4.2 — Done ✅ (canvas React Flow + `saveWorkflowCanvas` + schema Tasks/Steps funcional)
- **Requires:** Story 1.4 — Done ✅ (tabla `workflow_templates` en schema con `tasks_config` JSONB + seed de 8 templates)
- **Blocks:** Story 4.4 (Squads asignados a workflows — el `squad_type` del template se usa aquí)
- **Blocks:** Story 4.5 (Steps tipo `ai`/`human` procesados desde steps instanciados aquí)

## Scope

### IN

- **Template Gallery Modal/Panel:** UI para listar los 8 templates disponibles con nombre, categoría, descripción y `executor_type_default`
- **Server Action `getWorkflowTemplates()`:** Lista todos los templates del sistema (`is_system=true`) desde `workflow_templates`
- **Server Action `instantiateTemplate(workflowId, templateId)`:** Lee `tasks_config` del template y crea Tasks + Steps en DB + actualiza `canvas_data` del workflow con nodos y edges generados
- **Botón "Usar Template" en toolbar del canvas:** Abre el selector de templates (si el canvas está vacío o confirmando sobreescritura)
- **Instanciación:** Al elegir template, genera nodos React Flow (Tasks como rectángulos, Steps como círculos) con posiciones auto-layout (dagre LR) y los serializa como `canvas_data`
- **Confirmación de sobreescritura:** Si el canvas ya tiene nodos, muestra diálogo "¿Reemplazar canvas actual con este template?"
- **`templateId` en workflow:** Al instanciar, actualiza `workflows.template_id` con el UUID del template elegido
- **Indicador de template activo:** Muestra el nombre del template en el breadcrumb/toolbar si hay uno activo
- **Query layer:** `getWorkflowTemplates()` en `lib/db/queries/workflow-templates.ts`
- **Tests unitarios:** transformación `tasks_config → { nodes, edges }` (función pura sin DB)

### OUT

- Templates creados por el usuario (Story futura — `is_system=false`)
- Preview visual del template antes de instanciar (Phase 2)
- Edición de templates del sistema (solo lectura en MVP)
- Import/export de templates en formato YAML (Phase 2)
- Múltiples templates por proyecto (MVP = 1 por proyecto)
- Drag & drop del template al canvas (Phase 2)

## Acceptance Criteria

**AC1 — Botón "Usar Template" en toolbar:**
Given el usuario está en el canvas de workflow,
When ve el toolbar superior,
Then hay un botón "Usar Template" junto a "Add Task", "Add Step", "Auto-layout".

**AC2 — Lista de 8 templates:**
Given el usuario hace click en "Usar Template",
When el modal/panel se abre,
Then muestra los 8 templates del sistema con: nombre, categoría, descripción y executor_type_default (ícono human/ai/mixed).

**AC3 — Instanciación en canvas vacío:**
Given el canvas está vacío (sin nodos),
When el usuario selecciona un template y confirma,
Then el canvas muestra los Tasks y Steps del template con auto-layout LR, coloreados por executor_type, y el workflow se guarda automáticamente.

**AC4 — Confirmación de sobreescritura:**
Given el canvas ya tiene nodos,
When el usuario selecciona un template,
Then aparece un diálogo de confirmación "El canvas actual se reemplazará. ¿Continuar?".
When el usuario confirma,
Then el canvas se reemplaza con los nodos del template.

**AC5 — `templateId` persistido:**
Given el usuario instanció un template,
When el workflow se guarda,
Then `workflows.template_id` tiene el UUID del template elegido.

**AC6 — Indicador de template activo:**
Given un workflow tiene `template_id` no nulo,
When el usuario está en el canvas,
Then el toolbar o breadcrumb muestra el nombre del template activo (ej. "Template: Product Launch").

**AC7 — Steps coloreados por executor_type:**
Given el template tiene steps con executor_type 'human', 'ai', 'mixed',
When se instancia el template,
Then los nodos Step tienen el color correcto: human=azul, ai=púrpura, mixed=degradado.

**AC8 — Reload conserva template:**
Given el usuario instanció un template y recargar la página,
When `/projects/[id]/workflow` carga,
Then el canvas restaura exactamente los mismos nodos y edges del template instanciado.

**AC9 — Query de templates autenticada:**
Given el usuario está autenticado,
When `getWorkflowTemplates()` se llama,
Then retorna solo los templates `is_system=true` (sistema), sin requerir `user_id` en el filtro (templates globales).

**AC10 — Tests unitarios de transformación:**
Given la función `templateConfigToCanvas(tasksConfig, templateId)`,
When se le pasa un `tasks_config` con 2 tasks y 5 steps,
Then retorna `{ nodes, edges }` con 7 nodos (2 Task + 5 Step) y edges Task→Step correctos, sin dependencia de DB.

### Hallazgos

#### ⚠ CONCERN (No bloquea, se documenta como deuda técnica)

**C1 — Performance: Upsert one-by-one en saveWorkflowCanvas**
- **Archivo:** `actions/workflows.ts` ln 201-221 (tasks) y 241-271 (steps)
- **Descripción:** Las inserciones de tasks y steps se hacen en loops individuales (N queries por N nodos) en lugar de batch inserts.
- **Impacto:** Menor en MVP con templates de ~20 nodos. Degrada con plantillas grandes.
- **Recomendación:** Migrar a `db.insert(tasks).values([...all])` en Story 4.x posterior.

**C2 — Performance: O(n²) lookup de parent edge en WorkflowCanvas**
- **Archivo:** `components/workflow/WorkflowCanvas.tsx` ln 183-191
- **Descripción:** `edges.find()` con `nodes.find()` anidados al preparar stepSyncData.
- **Impacto:** Menor en MVP (<50 nodos). Escala cuadráticamente.
- **Recomendación:** Pre-indexar nodes en Map al inicio de handleSave.

**C3 — setTimeout sin cleanup en WorkflowCanvas**
- **Archivo:** `components/workflow/WorkflowCanvas.tsx` ln 212-213 y 272-273
- **Descripción:** `setTimeout(() => setSaveSuccess(false), 2000)` sin limpiar en unmount.
- **Impacto:** Memory leak si el usuario navega antes de 2 segundos.
- **Recomendación:** Usar `useRef` + `clearTimeout` en cleanup de useEffect.

**C4 — Race condition de baja probabilidad en TemplateGallery**
- **Archivo:** `components/workflow/TemplateGallery.tsx` ln 32-38
- **Descripción:** Usuario puede hacer click rápido en 2 templates mientras el diálogo está abierto; `pendingTemplate` se sobrescribe.
- **Impacto:** UX marginal. No causa corrupción de datos.
- **Recomendación:** Agregar flag `isConfirming` para deshabilitar clics durante confirmación.

**C5 — Falta de índice stepsActivities(taskId)**
- **Archivo:** `lib/db/schema/steps-activities.ts`
- **Descripción:** La query de eliminación de steps por taskId (ln 284-291 de workflows.ts) puede hacer full-scan.
- **Impacto:** Menor en MVP. Se nota con workflows con 100+ steps.
- **Recomendación:** Agregar índice en historia de optimización de DB.

#### ✅ Verificaciones adicionales

- **crypto.randomUUID():** Confirmado disponible como global en Node.js v24 + Next.js. No requiere import explícito. El análisis automatizado generó falso positivo en este punto.
- **Seguridad:** Ownership check correcto en `instantiateTemplate` (ln 373-379). Sin horizontal privilege escalation posible.
- **TypeScript:** Los dobles cast `as unknown as` son aceptables dado que FlowNode y @xyflow/react Node son estructuralmente compatibles pero tipados de forma diferente. No constituyen bug.
- **Tests:** 26 tests unitarios de `templateConfigToCanvas` con cobertura completa de casos edge. Mock de `applyDagreLayout` es técnicamente correcto para aislar la lógica pura.
