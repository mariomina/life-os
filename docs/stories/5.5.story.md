## Story

**As a** usuario que planifica su año,
**I want** ver en la Vista Año mis actividades reales del año completo como heatmap de 365 días,
**so that** pueda tener visibilidad de mi carga anual de un vistazo e identificar días de alta y baja actividad sin navegar mes a mes.

## Acceptance Criteria

### AC1 — Query `getActivitiesForYear`

**Given** un usuario autenticado y una fecha cualquiera del año,
**When** se llama `getActivitiesForYear(userId, date)`,
**Then:**
- Devuelve todas las `steps_activities` del usuario donde `scheduledAt` cae dentro del año calendario (1 enero 00:00:00.000Z – 31 diciembre 23:59:59.999Z) que contiene `date`
- Usa UTC arithmetic directo (mismo patrón que `getActivitiesForMonth`) para calcular los límites: `Date.UTC(year, 0, 1, 0, 0, 0, 0)` y `Date.UTC(year, 11, 31, 23, 59, 59, 999)`
- Incluye `areaName` (LEFT JOIN `areas`) y `habitTitle` (LEFT JOIN `habits`) — `null` si no aplica
- Incluye `areaColor` derivado de `areas.maslowLevel` usando `maslowLevelToColor()` (reutiliza de `calendar.ts`)
- Ordena por `scheduledAt ASC`
- Está ubicada en `lib/db/queries/calendar.ts` (junto a `getActivitiesForDay`, `getActivitiesForWeek`, `getActivitiesForMonth`)
- Comparte la misma interfaz `ActivityForCalendar` — sin nuevo tipo

### AC3 — YearView muestra heatmap con datos reales

**Given** el calendario muestra la Vista Año,
**When** hay actividades en `steps_activities` para ese año,
**Then:**
- Cada celda del grid de 365 días muestra un indicador visual de intensidad basado en el número de eventos de ese día:
  - 0 eventos → sin indicador (celda vacía, color muted)

- Al navegar a años anteriores/siguientes, el `CalendarClient` muestra los eventos del `events` array ya provisto (sin nueva query — se re-fetch solo si el usuario recarga la página)

### Patrón de query (referencia: `getActivitiesForMonth` en `lib/db/queries/calendar.ts`)

```typescript
// lib/db/queries/calendar.ts — agregar debajo de getActivitiesForMonth
export async function getActivitiesForYear(
  userId: string,
  date: Date
): Promise<ActivityForCalendar[]> {
  assertDatabaseUrl()

  // Compute year boundaries in UTC.
  // Year: Jan 1 00:00:00.000Z – Dec 31 23:59:59.999Z
  // Using Date.UTC() directly — avoid date-fns local timezone drift (gotcha de Story 5.3)
  const year = date.getUTCFullYear()

  const rangeStart = new Date(Date.UTC(year, 0, 1, 0, 0, 0, 0))    // Jan 1
  const rangeEnd = new Date(Date.UTC(year, 11, 31, 23, 59, 59, 999)) // Dec 31

  // Query idéntica a getActivitiesForMonth — solo cambian los límites
}
```

**IMPORTANTE:** `Date.UTC(year, 0, 1)` = 1 de enero (month 0-indexed). `Date.UTC(year, 11, 31)` = 31 de diciembre. No usar `startOfYear`/`endOfYear` de `date-fns` (opera en hora local del servidor).

### Tests — patrón (referencia: `isInMonth` en `calendar-queries.test.ts`)

```typescript
/** Mimics the year-range filter applied in getActivitiesForYear (UTC) */
function isInYear(scheduledAt: Date, date: Date): boolean {
  const year = date.getUTCFullYear()
  const rangeStart = new Date(Date.UTC(year, 0, 1, 0, 0, 0, 0))
  const rangeEnd = new Date(Date.UTC(year, 11, 31, 23, 59, 59, 999))
  return scheduledAt >= rangeStart && scheduledAt <= rangeEnd
}

// TARGET_YEAR_DATE = new Date('2024-06-15T00:00:00Z') → año 2024
// Casos: Jan 1 00:00:00Z included, Dec 31 23:59:59.999Z included,
//        Dec 31 23:59:59.999Z del 2023 excluded, Jan 1 00:00:00Z del 2025 excluded
// Caso extra (opcional): 2024 bisiesto → Feb 29 23:59:59.999Z included
```
