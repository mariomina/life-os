## Story

**As a** usuario que gestiona un workflow con steps asignados a un squad,
**I want** que al guardar el canvas los steps tipo `ai` encolen automáticamente su ejecución en AIOS y los steps tipo `human` se calendaricen automáticamente con una fecha/hora sugerida,
**so that** el workflow avance sin que yo tenga que intervenir manualmente en cada step — el sistema sabe quién ejecuta qué y cuándo.

## Dependencies

- **Requires:** Story 4.4 — Done ✅ (`workflows.squad_type` + `aiAgent` en steps AI persistido en DB)
- **Requires:** Story 4.2 — Done ✅ (`saveWorkflowCanvas` sincroniza steps en `steps_activities`)
- **Blocks:** Story 4.6 (Hábitos — usa el mismo mecanismo de calendarización de steps)
- **Blocks:** Story 8.7 (Agent Leverage Report — necesita steps AI con estado de cola)

## Scope

### IN

- **Procesamiento post-save de steps AI:** Al llamar `saveWorkflowCanvas()`, los steps con `executorType='ai'` y `status='pending'` se procesan como "encolados en AIOS" — se actualiza su `status` a `'in_progress'` y se registra un log en `aios_queue_log` (tabla nueva)
- **Calendarización de steps Human:** Al llamar `saveWorkflowCanvas()`, los steps con `executorType='human'` y `status='pending'` reciben un `scheduledAt` sugerido (primer slot disponible a partir de mañana, 09:00, con duración estimada de 30 min por defecto)
- **Tabla `aios_queue_log`:** Nueva tabla con campos: `id`, `step_id`, `user_id`, `agent`, `status` (`queued | running | completed | failed`), `queued_at`, `completed_at`, `result_summary`, `created_at`
- **Server Action `processWorkflowSteps(workflowId)`:** Llamada automáticamente desde `saveWorkflowCanvas()` después de sincronizar steps. Itera steps pending y aplica la lógica AI/human
- **Indicador visual en StepNode:** Nodos AI muestran badge de estado de cola (`En cola`, `En progreso`, `Completado`, `Error`) leído desde el canvas data; nodos Human muestran la fecha calendarizada si existe (`scheduledAt`)
- **Query layer:** `createAiosQueueEntry()` y `getStepQueueStatus()` en `lib/db/queries/`
- **Tests unitarios:** lógica de selección de slot de calendarización (función pura), mapeo de status de cola
- **Migration:** nueva tabla `aios_queue_log` con RLS

### OUT

- Integración real con API de agentes AIOS externos (MVP: solo registro en cola local)
- Ejecución real de steps por agentes (Phase 2)
- Notificaciones push/email cuando un step AI completa
- Integración con calendario externo (Google Calendar, Apple Calendar)
- UI dedicada para gestión de la cola AIOS (solo badge en nodo del canvas en MVP)
- Resolución de conflictos de horario en calendarización (MVP: slot fijo 09:00 del día siguiente hábil)
- Steps tipo `mixed` (se tratan como `human` en MVP)

## Acceptance Criteria

**AC1 — Steps AI se encolan al guardar:**
Given el usuario guarda el canvas con steps `executorType='ai'` en estado `pending`,
When `saveWorkflowCanvas()` completa sin error,
Then cada step AI tiene un registro en `aios_queue_log` con `status='queued'` y el `agent` del step (`aiAgent`), y el step en `steps_activities` pasa a `status='in_progress'`.

**AC2 — Steps Human reciben scheduledAt al guardar:**
Given el usuario guarda el canvas con steps `executorType='human'` en estado `pending`,
When `saveWorkflowCanvas()` completa sin error,
Then cada step Human tiene `scheduledAt` asignado (primer día hábil siguiente a las 09:00 UTC) y `scheduledDurationMinutes=30` en `steps_activities`.

**AC3 — Solo steps pending se procesan:**
Given el canvas tiene steps en estado `in_progress`, `completed`, `skipped` o `cancelled`,
When se llama `saveWorkflowCanvas()`,
Then esos steps NO son re-procesados (no se crea nuevo log, no se modifica `scheduledAt`).

**AC4 — Tabla `aios_queue_log` tiene RLS:**
Given la tabla `aios_queue_log` existe en Supabase,
When se consulta desde cualquier cliente,
Then solo el usuario propietario (`user_id`) puede leer/escribir sus propios registros.

**AC5 — Badge de estado en nodo AI:**
Given un step `executorType='ai'` tiene un entry en `aios_queue_log`,
When el canvas carga (datos del workflow),
Then el `StepNode` muestra el badge de estado de cola (`En cola` / `En progreso` / `Completado` / `Error`) según el `status` del último log entry.

**AC6 — Badge de fecha en nodo Human:**
Given un step `executorType='human'` tiene `scheduledAt` asignado en DB,
When el canvas carga,
Then el `StepNode` muestra la fecha calendarizada en formato legible (ej. `"Mañana 09:00"` o `"Lun 09:00"`).

**AC7 — Steps mixtos se tratan como human:**
Given un step con `executorType='mixed'`,
When se guarda el canvas,
Then recibe `scheduledAt` como si fuera `human` (no se encola en AIOS).

**AC8 — `processWorkflowSteps` es idempotente:**
Given `saveWorkflowCanvas()` es llamado dos veces seguidas sin cambios en los steps,
When la segunda llamada procesa steps,
Then no se crean entradas duplicadas en `aios_queue_log` y no se modifica `scheduledAt` existente.

**AC9 — Tests unitarios de calendarización:**
Given la función pura `getNextBusinessDaySlot(fromDate: Date): Date`,
When se llama con cualquier fecha (incluyendo viernes y fines de semana),
Then retorna el siguiente día hábil (lunes-viernes) a las 09:00 UTC.

**AC10 — Sin regresión en save existente:**
Given el canvas tiene tasks y steps sin cambios de executor_type,
When el usuario guarda el canvas múltiples veces,
Then el comportamiento previo de `saveWorkflowCanvas()` (upsert tasks, upsert steps, save canvas_data) funciona correctamente — 243 tests previos siguen pasando.

### Nueva tabla `aios_queue_log`

```typescript
// lib/db/schema/aios-queue-log.ts (archivo nuevo)
export const aiosQueueLog = pgTable('aios_queue_log', {
  id: uuid('id').defaultRandom().primaryKey(),
  stepId: uuid('step_id')
    .notNull()
    .references(() => stepsActivities.id, { onDelete: 'cascade' }),
  userId: uuid('user_id').notNull(),

  agent: text('agent').notNull(),
  /** Estado de la tarea en cola */
  status: text('status')
    .$type<'queued' | 'running' | 'completed' | 'failed'>()
    .default('queued')
    .notNull(),
  queuedAt: timestamp('queued_at', { withTimezone: true }).defaultNow().notNull(),
  completedAt: timestamp('completed_at', { withTimezone: true }),
  /** Resumen del resultado cuando el agente completa (Phase 2) */
  resultSummary: text('result_summary'),
  createdAt: timestamp('created_at', { withTimezone: true }).defaultNow().notNull(),
})
```

### Migration SQL

```sql
-- aios_queue_log
CREATE TABLE aios_queue_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  step_id UUID NOT NULL REFERENCES steps_activities(id) ON DELETE CASCADE,
  user_id UUID NOT NULL,
  agent TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'queued'
    CHECK (status IN ('queued', 'running', 'completed', 'failed')),
  queued_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMPTZ,
  result_summary TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

ALTER TABLE aios_queue_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY "aios_queue_log_owner" ON aios_queue_log
  FOR ALL USING (auth.uid() = user_id);

CREATE INDEX aios_queue_log_step_id_idx ON aios_queue_log(step_id);
CREATE INDEX aios_queue_log_user_status_idx ON aios_queue_log(user_id, status);
```

### Función pura de calendarización

```typescript
// lib/workflow/scheduling-utils.ts (archivo nuevo)

/**
 * Retorna el siguiente día hábil (lun-vie) a las 09:00 UTC
 * a partir de la fecha dada (siempre avanza al menos 1 día).
 */
export function getNextBusinessDaySlot(fromDate: Date): Date {
  const next = new Date(fromDate)
  next.setUTCDate(next.getUTCDate() + 1) // siempre al menos mañana
  // Avanzar hasta lunes-viernes
  while (next.getUTCDay() === 0 || next.getUTCDay() === 6) {
    next.setUTCDate(next.getUTCDate() + 1)
  }
  next.setUTCHours(9, 0, 0, 0)
  return next
}
```

### Server Action `processWorkflowSteps`

```typescript
// actions/workflows.ts (agregar)
// Llamado internamente por saveWorkflowCanvas después del upsert de steps

async function processWorkflowSteps(
  workflowId: string,
  userId: string,
  stepIds: string[]
): Promise<void>
```

Lógica:
1. Leer steps de `stepIds` desde `steps_activities` WHERE `status='pending'`
2. Para cada step `executorType='ai'`:
   - Verificar que NO existe ya un log entry para ese `step_id` con `status IN ('queued','running')` — idempotencia
   - Insertar en `aios_queue_log` con `status='queued'`, `agent=step.aiAgent`
   - UPDATE `steps_activities SET status='in_progress'` para ese step
3. Para cada step `executorType='human'` o `'mixed'`:
   - Verificar que `scheduledAt IS NULL` — idempotencia
   - UPDATE `steps_activities SET scheduled_at=getNextBusinessDaySlot(NOW()), scheduled_duration_minutes=30`

### Badge de estado en StepNode

`StepNode` ya recibe `data.aiAgent` y `data.executorType`. Para el badge de cola, el `canvas_data` almacenado debe incluir el `queueStatus` en `data` del nodo. Al cargar el canvas en `page.tsx`, el Server Component enriquece los nodos AI con el último `status` de `aios_queue_log`.

Alternativamente (más simple para MVP): mostrar estado directamente desde `step.status` de `steps_activities`:
- `pending` → sin badge
- `in_progress` → "En cola" (amarillo)
- `completed` → "Completado" (verde)
- `cancelled` / `failed` → "Error" (rojo)
