## Story

**As a** developer working on life-os,
**I want** a GitHub Actions CI pipeline that runs lint, typecheck, and build on every PR and push to main, along with branch protection rules enforced on `main` and Supabase secrets configured as GitHub repository secrets,
**so that** no broken code can be merged to the main branch and the team has a reliable automated quality gate from which all future features can be developed with confidence.

## Dependencies

- **Requires:** Story 1.2 — Status: Done ✅ (Supabase project exists with real URL and anon key; auth layer functional; `npm run build`, `npm run typecheck`, and `npm run lint` all pass)
- **Blocks:** Story 1.5+ (all future stories benefit from CI gate on PRs); Story 1.6 (Vitest tests will be added to the CI pipeline once the testing framework is set up)

## Scope

**IN scope:**
- `.github/workflows/ci.yml` — GitHub Actions workflow triggered on `push` and `pull_request` targeting `main`
- CI steps: install dependencies → lint (`npm run lint`) → typecheck (`npm run typecheck`) → build (`npm run build`)
- GitHub repository secrets: `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` configured so the build step can resolve env vars
- Branch protection rule on `main`: require status checks to pass before merging, require PR (no direct push)
- Optional: CI status badge in `README.md`

**OUT of scope:**
- Vercel deployment pipeline (Vercel handles its own build/deploy integration separately)
- Vitest test step in CI — Story 1.6 (@dev) installs the testing framework; CI test step added then
- Supabase migrations in CI (no automated `drizzle-kit push` or `supabase db push` in pipeline for MVP)
- Docker-based Supabase local in CI (`supabase start`) — uses cloud project credentials for build only
- E2E tests (Playwright) — deferred to Phase 2 per test strategy
- Secrets rotation or GitHub Environments (staging/production) — post-MVP

## Complexity

**Estimacion:** S — 2 story points
- Razon: GitHub Actions YAML is well-documented and the commands already work locally. The main risk is env var configuration for the build step. Branch protection is a UI operation in GitHub settings.

## Risks

| Riesgo | Probabilidad | Impacto | Mitigacion |
|--------|-------------|---------|------------|
| `npm run build` falla en CI por env vars faltantes | Media | Alto | Configurar `NEXT_PUBLIC_SUPABASE_URL` y `NEXT_PUBLIC_SUPABASE_ANON_KEY` como GitHub Secrets; usar valores reales del proyecto Supabase cloud |
| `SUPABASE_SERVICE_ROLE_KEY` expuesta en CI logs | Baja | Critico | Nunca agregar `SERVICE_ROLE_KEY` como secret publico en CI; solo las dos variables `NEXT_PUBLIC_*` son necesarias para el build |
| Branch protection bloquea workflow de @devops | Baja | Medio | Configurar la regla para que los status checks de CI sean el requisito, no bloquear force-push de emergencia para @devops |
| Cache de `node_modules` inconsistente entre runs | Baja | Bajo | Usar `actions/cache` con key basada en `package-lock.json` hash |
| Tiempo de CI excesivo (>5 min) | Baja | Bajo | Cache de dependencias + paralelizar lint/typecheck si es necesario |

## Acceptance Criteria

1. Archivo `.github/workflows/ci.yml` creado — workflow se activa en `push` a cualquier rama y en `pull_request` targeting `main`
2. El workflow ejecuta los pasos en orden: checkout → setup Node.js 20 → cache `node_modules` → `npm ci` → `npm run lint` → `npm run typecheck` → `npm run build` — todos deben pasar para que el check sea verde
3. GitHub repository secrets configurados: `NEXT_PUBLIC_SUPABASE_URL` y `NEXT_PUBLIC_SUPABASE_ANON_KEY` con los valores reales del proyecto Supabase cloud (verificar que `npm run build` en CI no falla por env vars)
4. Branch protection rule habilitada en `main`: "Require status checks to pass before merging" con el check `ci` (o nombre del job) como required; "Require a pull request before merging" activado
5. Al crear un PR de prueba con un cambio trivial, el check de CI aparece en la PR de GitHub y muestra estado "passing" o "failing" correctamente
6. Si `npm run lint` falla (introducir error intencional y revertir), el CI check se marca como failed y bloquea el merge — verificado y revertido
7. `README.md` actualizado con badge de CI (opcional pero recomendado): `![CI](https://github.com/{owner}/{repo}/actions/workflows/ci.yml/badge.svg)`

## Dev Notes

### Estructura del workflow ci.yml

[Source: docs/architecture/infrastructure.md#10.2, docs/architecture/test-strategy.md#13.5]

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: ['**']
  pull_request:
    branches: [main]

jobs:
  ci:
    name: Lint, Typecheck & Build
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Typecheck
        run: npm run typecheck

      - name: Build
        run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
          NEXT_PUBLIC_APP_URL: http://localhost:3000
```

### Por que solo las variables NEXT_PUBLIC_*

Next.js requiere que las variables `NEXT_PUBLIC_*` esten disponibles en build time para ser inlined en el bundle del cliente. El build fallara con `undefined` si no estan presentes. La `SUPABASE_SERVICE_ROLE_KEY` y `DATABASE_URL` son solo server-side y Next.js no las necesita en build time para este proyecto (no hay queries DB en build-time, solo en runtime).

### Vitest en CI — no incluido en esta story

[Source: docs/architecture/test-strategy.md#13.5]

El paso `npm test` (Vitest) se agregara al workflow en Story 1.6 cuando el testing framework quede instalado. En esta story el CI solo cubre lint + typecheck + build, que es suficiente para el estado actual del proyecto.

### Branch protection — consideraciones para @devops

La regla de branch protection debe configurarse de modo que @devops (admin del repo) pueda hacer force-push o bypass en situaciones de emergencia (hotfix critico). La opcion "Allow force pushes" debe quedar en "Specify who can force push" → agregar al admin. Esto es diferente a "Do not allow bypassing" que aplica a todos.

### Archivos a crear en esta story

- `.github/workflows/ci.yml` — workflow principal
- `README.md` — agregar badge (si existe, modificar; si no existe, crear minimal)

### Archivos a NO modificar en esta story

- Cualquier archivo de la aplicacion (src, lib, app, etc.)
- `package.json` scripts (ya estan correctos: lint, typecheck, build)
- Supabase configuration

### Verificacion local previa al push

Antes de hacer push del workflow, verificar localmente que los tres comandos pasan:

```bash
npm run lint       # debe pasar con 0 errores
npm run typecheck  # debe pasar con 0 errores
npm run build      # puede mostrar warning de middleware deprecado pero debe completar
```
