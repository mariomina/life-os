## Story

**I want** Vitest and React Testing Library configured with proper mocks for Supabase and Next.js, plus the first unit tests covering the Maslow scoring formula and maslow-weights constants,
**so that** every subsequent story has a working test harness and the most critical pure-logic functions are validated from day one.

## Scope

**IN scope:**
- Instalación de Vitest 3.x + `@vitejs/plugin-react` + `@testing-library/react` + `@testing-library/user-event` + `@testing-library/jest-dom`
- Configuración de `vitest.config.ts` compatible con Next.js App Router (jsdom environment, path aliases `@/`, TS paths)
- Setup global de tests: `tests/setup.ts` con `@testing-library/jest-dom` matchers
- Mock de `next/navigation` (usePathname, useRouter, redirect) para componentes Next.js
- Mock de Supabase client (`@/lib/supabase/client`) para tests unitarios sin DB real
- Script `"test"` en `package.json`: `vitest`
- Script `"test:coverage"` en `package.json`: `vitest --coverage`
- Creación de `features/maslow/scoring.ts` con funciones `calculateAreaScore` y `calculateGlobalScore` (código puro, sin dependencias externas)
- Tests unitarios para `features/maslow/scoring.ts` — cobertura de casos base, multiplicadores Maslow y edge cases
- Tests unitarios para `lib/utils/maslow-weights.ts` — verificar constantes y suma de pesos
- Integración con CI: `npm test -- --run` en `ci.yml` (modo non-interactive, sin watch)
- Cobertura mínima 80% en los archivos creados en esta story

**OUT of scope:**
- Tests de integración contra Supabase local (Docker) — stories de features específicas los añaden
- E2E con Playwright — Phase 2
- Tests de componentes UI (Header.tsx, Sidebar.tsx) — scope de story de mejoras de UI
- Coverage gate en CI bloqueante — se activa en Story 2.x cuando haya más código
- Supabase local setup (`supabase start`) — no requerido para tests unitarios

## Complexity

**Estimación:** S — 3 story points
- Razón: Setup de Vitest con Next.js tiene particularidades conocidas (path aliases, jsdom, mocks de next/navigation), pero es configuración estándar. Las funciones de scoring son código puro fácil de testear. El mayor riesgo es la compatibilidad de Vitest con el módulo system de Next.js 16.

## Risks

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Vitest incompatible con Next.js 16 Turbopack | Baja | Medio | Usar `vitest.config.ts` independiente de Vite/Turbopack — Vitest no requiere Vite en el build de Next.js, solo para los tests |
| Path aliases `@/` no resuelven en test environment | Media | Medio | Configurar `resolve.alias` en `vitest.config.ts` usando `tsconfig.json` paths |
| `useRouter` / `usePathname` fallan en tests | Alta | Bajo | Mock estándar de `next/navigation` documentado en Next.js testing guide |
| `@testing-library/jest-dom` matchers no disponibles | Baja | Bajo | Import en `tests/setup.ts` y referenciar en `vitest.config.ts` setupFiles |

1. `npm test` ejecuta la suite de Vitest y pasa sin errores — todos los tests en verde

5. Mock de `next/navigation` disponible — tests de componentes que usen `usePathname` o `useRouter` no fallan por módulo no implementado
6. `features/maslow/scoring.ts` creado con dos funciones exportadas:
   - `calculateAreaScore(rawScore: number, maslowLevel: MaslowLevel): number` — aplica multiplicador del nivel
   - `calculateGlobalScore(areaScores: Record<MaslowLevel, number>): number` — calcula Life System Health Score ponderado (suma(score × multiplicador) / 11.4)
7. `tests/unit/maslow/scoring.test.ts` con mínimo 8 casos de test que cubran:
   - Score cero en todas las áreas → global = 0
   - Score 100 en todas las áreas → global = 100
   - Solo D-Needs (niveles 1-4) al 100%, B-Needs al 0% → global ≈ 61.4%
   - Multiplicadores correctos para cada nivel (2.0×, 1.5×, 1.2×, 1.0×)

   - `MASLOW_TOTAL_WEIGHT === 11.4`

9. CI (`ci.yml`) actualizado para ejecutar `npm test -- --run` (modo non-watch) después de typecheck y antes del lint
10. `npm run typecheck` pasa sin errores tras todos los cambios

### Vitest con Next.js 16 — Patrón de configuración

[Source: docs/architecture/test-strategy.md#13.2]

Vitest 3.x funciona independientemente del build system de Next.js. El `vitest.config.ts` es un archivo separado que solo se usa al ejecutar `npm test`. NO modificar `next.config.ts` ni `postcss.config.ts`.

**Clave:** Los path aliases `@/` del `tsconfig.json` NO se resuelven automáticamente en Vitest — hay que duplicarlos en `vitest.config.ts` bajo `resolve.alias`.

### Mock de next/navigation

[Source: Next.js Testing Docs]

Para componentes que usen `usePathname`, `useRouter` o `redirect`, crear mock manual en `tests/__mocks__/next/navigation.ts`:

```typescript
// tests/__mocks__/next/navigation.ts
import { vi } from 'vitest'

export const usePathname = vi.fn(() => '/')
export const useRouter = vi.fn(() => ({
  push: vi.fn(),
  replace: vi.fn(),
  back: vi.fn(),
  forward: vi.fn(),
  refresh: vi.fn(),
  prefetch: vi.fn(),
}))
export const redirect = vi.fn()
export const useSearchParams = vi.fn(() => new URLSearchParams())
```

Para que Vitest use este mock automáticamente, añadir en `vitest.config.ts`:
```typescript
alias: {
  'next/navigation': path.resolve(__dirname, 'tests/__mocks__/next/navigation.ts'),
}
```

### Función calculateGlobalScore — fórmula exacta

[Source: docs/prd/requirements.md#FR1 + docs/prd/technical-assumptions.md]

```typescript
// La fórmula del Life System Health Score:
// Score_global = Σ(score_área × multiplicador_nivel) / Σ(multiplicadores)
// Σ(multiplicadores) = MASLOW_TOTAL_WEIGHT = 11.4

// Ejemplo: D-Needs al 100%, B-Needs al 0%
// = (100×2.0 + 100×2.0 + 100×1.5 + 100×1.5 + 0×1.2 + 0×1.2 + 0×1.0 + 0×1.0) / 11.4
// = (200 + 200 + 150 + 150) / 11.4
// = 700 / 11.4
// ≈ 61.40%
```

### Cobertura — qué medir y qué no

[Source: docs/architecture/test-strategy.md#13.1]

- **NO testear en esta story:** Componentes UI (Sidebar, Header), layouts de Next.js, configuración de Supabase

Vitest coverage con `@vitest/coverage-v8` genera reporte HTML en `coverage/` — añadir `coverage/` a `.gitignore`.

El `ci.yml` actual tiene estos steps: `checkout → setup-node → npm ci → typecheck → lint → build`. Añadir `test:run` entre `typecheck` y `lint`:

```yaml
- name: Run tests

```

### Testing Standards acordados

[Source: docs/architecture/test-strategy.md]

- **Naming:** `*.test.ts` en `tests/unit/` espejando la estructura de `features/` y `lib/`
- **Mocking:** `vi.mock()` para módulos externos — no mockear código propio
- **Assertions:** Usar `@testing-library/jest-dom` matchers — `expect(x).toBe(y)` para valores primitivos
- **No snapshot tests** — frágiles y difíciles de mantener
