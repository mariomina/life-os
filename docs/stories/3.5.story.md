## Story

**As a** usuario que quiere alinear su vida con una visión de largo plazo,
**I want** crear una Visión narrativa de 5 años y hasta 3 OKRs anuales vinculados a mis áreas de vida,
**so that** cada objetivo anual esté anclado a mi dirección estratégica y el sistema me impida dispersarme con más de 3 prioridades anuales (regla Buffett 5/25).

## Scope

**IN scope:**
- Query layer `lib/db/queries/okrs.ts`:
  - `getVision(userId)` → OKR de tipo `vision` del usuario (único, o null)
  - `getAnnualOKRs(userId, year)` → OKRs de tipo `annual` del año dado
  - `countActiveAnnualOKRs(userId, year)` → entero, para validar regla ≤3
- Server Actions `actions/okrs.ts`:
  - `upsertVision(userId, data)` — crear/actualizar narrativa de visión (sin área)
  - `createAnnualOKR(userId, data)` — crear OKR anual con validación Buffett (≤3 activos)
  - `updateAnnualOKR(userId, id, data)` — actualizar título/descripción/área/year de OKR anual
  - `deleteOKR(userId, id)` — delete lógico (status='cancelled') — sin borrado físico
- Página Server Component `app/(app)/okrs/page.tsx`:
  - Carga `vision` + `annualOKRs` del año actual en paralelo con `Promise.all`
  - Renderiza sección Visión + sección OKRs anuales
  - Pasa datos como props a los Client Components (sin fetch client-side)
- Componente `components/okrs/VisionCard.tsx` (Client Component):
  - Muestra narrativa de visión actual
  - Botón "Editar visión" abre textarea en línea (sin modal)
  - Llama `upsertVision` action al guardar
- Componente `components/okrs/AnnualOKRList.tsx` (Client Component):
  - Lista de OKRs anuales del año actual con: título, área vinculada, progreso (barra), estado
  - Muestra `OKRImpactBadge` (reutilizado de Story 3.4)
  - Botón "Nuevo OKR" deshabilitado si ya hay 3 activos (tooltip: "Máx. 3 OKRs anuales activos — Buffett 5/25")
  - Integra `OKRForm` (de Story 3.3) para creación con `type='annual'`
- Validación Buffett 5/25 en Server Action: si `countActiveAnnualOKRs >= 3` → error 422 con mensaje claro
- RLS: todas las queries usan `userId` del contexto de auth de Supabase; no se expone data de otros usuarios
- Tests unitarios en `tests/unit/okrs/`:
  - `buffett-rule.test.ts` — validación de la regla ≤3 OKRs (pura lógica, sin DB)
  - `okr-queries.test.ts` — tests de las funciones query con mock de Drizzle
- TypeScript strict, sin `any`

**OUT of scope:**
- KRs trimestrales (Story 3.6)
- Progreso automático de OKRs (Story 3.7)
- Drag & drop para reordenar OKRs
- Soft-delete UI (cancelación se hace via edición de estado, no se muestra en MVP)
- Historial de versiones de la visión
- Multi-año (siempre muestra el año actual `new Date().getFullYear()`)

## Acceptance Criteria

### AC1 — Visión narrativa (CRUD)
**Given** un usuario autenticado en `/okrs`,
**When** no tiene visión guardada,
**Then** se muestra un estado vacío con CTA "Define tu visión de 5 años" y un textarea para escribir la narrativa.

**Given** el usuario escribe y guarda una visión,
**When** recarga la página,
**Then** la visión persiste y se muestra en la sección Visión.

**Given** el usuario edita la visión existente y guarda,
**When** la acción `upsertVision` se ejecuta,
**Then** el registro en `okrs` con `type='vision'` se actualiza (no se crea duplicado).

### AC2 — Listado de OKRs anuales
**Given** el usuario tiene OKRs anuales del año actual,
**When** accede a `/okrs`,
**Then** ve la lista con: título, nombre del área vinculada, barra de progreso (siempre 0% en esta historia — Story 3.7 lo calculará), badge de estado (`active`), y `OKRImpactBadge` si `deltaPoints > 0`.

### AC3 — Regla Buffett 5/25 (máx 3 OKRs activos)
**Given** el usuario ya tiene 3 OKRs anuales con `status='active'` para el año actual,
**When** intenta crear un 4to OKR anual,
**Then**:
- El botón "Nuevo OKR" está deshabilitado en la UI con tooltip explicativo.
- Si la petición llega al Server Action de todos modos, retorna error 422: `"Máximo 3 OKRs anuales activos (regla Buffett 5/25). Cancela uno antes de crear otro."`
- No se crea ningún registro en DB.

**Given** el usuario tiene ≤2 OKRs activos,
**When** intenta crear uno nuevo,
**Then** el formulario se muestra y la creación procede normalmente.

### AC4 — Jerarquía OKR correcta en DB
**Given** el usuario crea un OKR anual,
**When** la acción `createAnnualOKR` se ejecuta,
**Then** el registro en `okrs` tiene:
- `type = 'annual'`
- `parent_id` = id del registro de visión del usuario (si existe), o `NULL` si aún no tiene visión
- `year` = año actual
- `status = 'active'`
- `progress = 0`
- `area_id` = UUID del área seleccionada (puede ser NULL si no se selecciona área)

### AC5 — Validación jerárquica heredada (Story 3.3)
**Given** el usuario intenta crear un OKR anual vinculado a un área de nivel 7-8,
**When** algún área de nivel 1-2 tiene score <50% por >14 días consecutivos,
**Then** el formulario muestra el `HierarchyWarning` (componente de Story 3.3) y el botón de submit está deshabilitado — idéntico al comportamiento ya implementado en `OKRForm`.

### AC6 — Vacío de OKRs anuales
**Given** el usuario no tiene OKRs anuales para el año actual,
**When** accede a `/okrs`,
**Then** ve un estado vacío en la sección OKRs con CTA "Crear primer OKR" y el formulario disponible.

## Dev Notes

### Arquitectura de la página

La página `app/(app)/okrs/page.tsx` es un **Server Component puro** — carga todos los datos en servidor y los pasa como props:

```typescript
// app/(app)/okrs/page.tsx (Server Component)
const [vision, annualOKRs, userAreas, scoreHistory] = await Promise.all([
  getVision(user.id),
  getAnnualOKRs(user.id, currentYear),
  getUserAreas(user.id),           // para OKRForm (hierarchy guard + impact)
  getRecentAreaScores(user.id, 30) // para OKRForm (hierarchy guard)
])
```

### Query layer — `lib/db/queries/okrs.ts`

```typescript
import { db } from '@/lib/db'
import { okrs } from '@/lib/db/schema/okrs'
import { eq, and, count } from 'drizzle-orm'

export async function getVision(userId: string): Promise<OKR | null>
export async function getAnnualOKRs(userId: string, year: number): Promise<OKR[]>
export async function countActiveAnnualOKRs(userId: string, year: number): Promise<number>
```

```typescript

```

### RLS en Supabase

La tabla `okrs` tiene RLS habilitada (Story 1.4). Las queries Drizzle deben incluir siempre el filtro `userId` explícito — no confiar solo en RLS para correctness del código.

### Progreso en esta historia

En Story 3.5, el campo `progress` siempre será `0` en los OKRs anuales recién creados. La barra de progreso en la UI lo refleja. El cálculo automático se implementa en Story 3.7.
