## Story

**As a** usuario que quiere hacer una revisión semanal estructurada,
**I want** un flujo guiado de 4 fases (Medir, Analizar, Planificar, Confirmar) que agregue automáticamente mis métricas y me lleve a comprometer la próxima semana,
**so that** el Weekly Review sea sistemático, rápido y basado en datos reales sin preparación manual.

## Acceptance Criteria

### AC1 — Página `/weekly-review` con flujo de 4 fases
**Given** el usuario navega a `/weekly-review`,
**When** se renderiza la página,
**Then** aparece un wizard de 4 pasos con indicador de progreso (paso 1/4, 2/4, etc.):
- Fase 1: Medir
- Fase 2: Analizar
- Fase 3: Planificar
- Fase 4: Confirmar

### AC2 — Fase 1: Medir (métricas automáticas)
**Given** el usuario está en Fase 1,
**When** se renderiza,
**Then** muestra automáticamente (sin input del usuario):
- CCR de la semana pasada (% actividades completadas)
- Habit Consistency promedio (%)
- OKR Progress promedio (%)
- Time by Area top 3 áreas de la semana
- Area Health Trends (mejorando/estable/deteriorando)
Botón "Siguiente →" para avanzar a Fase 2.

### AC3 — Fase 2: Analizar (insights IA + patrones)
**Given** el usuario está en Fase 2,
**When** se renderiza,
**Then** muestra:
- Insights en lenguaje natural de `generateReportInsights()` (de 8.6)
- Patrones detectados si existen (de 8.4): bucles, leverage, bottlenecks
- Si ILLMProvider no disponible: análisis descriptivo estático
Botón "Siguiente →" para avanzar a Fase 3.

### AC4 — Fase 3: Planificar (inbox + backlog review)
**Given** el usuario está en Fase 3,
**When** se renderiza,
**Then** muestra:
- Items del Inbox sin procesar (de `lib/db/queries/inbox.ts`)
- Activities pendientes (`status='pending'`) del backlog del usuario (top 10 por área)
- El usuario puede marcar items del inbox como "para la próxima semana" o "archivar"
Botón "Siguiente →" para avanzar a Fase 4.

### AC5 — Fase 4: Confirmar (semana comprometida)
**Given** el usuario está en Fase 4,
**When** se renderiza,
**Then** muestra:
- Resumen de la semana comprometida: activities ya programadas para los próximos 7 días
- Conteo: X activities planificadas, Y horas estimadas
- Botón "✓ Completar Weekly Review" que guarda la fecha del review en localStorage
- Mensaje de cierre: "¡Weekly Review completado! Próxima semana comprometida."

### AC6 — Server Action `getWeeklyReviewData()`
**Given** el usuario autenticado,
**When** se llama a `getWeeklyReviewData()`,
**Then** retorna en una sola llamada:
```typescript
{
  metrics: WeeklyMetrics        // CCR, habit consistency, OKR progress, time by area
  insights: string              // de ILLMProvider
  patterns: Correlation[]       // correlaciones activas con tipo != neutral
  inboxItems: InboxItem[]       // items sin procesar
  pendingActivities: StepActivity[]  // backlog top 10
  nextWeekActivities: StepActivity[] // próximos 7 días
}
```

### AC7 — Acceso rápido desde Home/sidebar
**Given** el usuario en cualquier pantalla,
**When** hace clic en "Weekly Review" en el sidebar,
**Then** navega a `/weekly-review`.

### AC8 — Persistencia del último review completado
**Given** el usuario completa un Weekly Review,
**When** hace clic en "Completar",
**Then** se guarda `weeklyReviewCompletedAt` en localStorage,
Y el Home/sidebar muestra "Último review: hace X días" como indicador.

### AC9 — Tests ≥ anteriores + nuevos de Story 8.8
**Given** la implementación completa,
**When** se ejecuta `npm test`,
**Then** todos los tests pasan.

## Business Value

Cierra FR12 completamente. El Weekly Review es el mecanismo de clausura semanal del sistema — sin él, las métricas de 8.1-8.7 son datos aislados. Con el wizard guiado, el usuario tiene un ritual sistemático de 15 minutos que le da visibilidad completa, insights IA y la semana próxima comprometida. Es el producto mínimo viable de "sistema de vida" funcional end-to-end.

## Risks

- **`getWeeklyReviewData` lenta:** Agregación de múltiples Server Actions. Usar `Promise.allSettled` para paralelizar y no bloquear si una falla.
- **Fase 3 inbox vacío:** Manejar estado vacío con mensaje motivador ("¡Inbox limpio! Nada pendiente de revisar.").
- **LLM timeout en Fase 2:** El fallback estático asegura que el wizard no se bloquea.

## Dev Notes

### `aggregateWeeklyMetrics` en `features/reports/weekly.ts`

```typescript
export interface WeeklyMetrics {
  ccrRate: number | null
  habitConsistencyAvg: number
  okrProgressAvg: number
  timeByAreaTop3: { areaName: string; totalSeconds: number }[]
  areaHealthTrends: { areaName: string; trend: 'improving' | 'declining' | 'stable' }[]
}

export function aggregateWeeklyMetrics(
  ccr: { planned: number; completed: number; rate: number | null },
  habits: { rate: number }[],
  okrs: { avgProgress: number }[],
  timeByArea: { areaName: string; totalSeconds: number }[],
  areaHealthTrends: { areaName: string; trend: 'improving' | 'declining' | 'stable' }[]
): WeeklyMetrics {
  const habitConsistencyAvg = habits.length > 0 ? habits.reduce((s, h) => s + h.rate, 0) / habits.length : 0
  const okrProgressAvg = okrs.length > 0 ? okrs.reduce((s, o) => s + o.avgProgress, 0) / okrs.length : 0
  return {
    ccrRate: ccr.rate,
    habitConsistencyAvg,
    okrProgressAvg,
    timeByAreaTop3: timeByArea.slice(0, 3),
    areaHealthTrends,
  }
}
```
