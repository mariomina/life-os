## Story

**As a** nuevo usuario de life-os que acaba de autenticarse por primera vez,
**I want** ser recibido por un flujo de onboarding claro que me explique brevemente el sistema y me permita elegir cómo quiero hacer mi diagnóstico inicial de las 8 áreas Maslow (cuestionario científico o upload de archivos psicométricos),
**so that** el sistema tenga el baseline de mi estado actual antes de mostrarme el Home con las áreas y el Life System Health Score.

## Scope

**IN scope:**
- Detección de usuario nuevo (sin `area_scores` previos en DB) vs. usuario existente → redirect al onboarding solo si es nuevo
- Ruta `/onboarding` en `app/(auth)/onboarding/page.tsx` (sin sidebar, layout auth)
- Pantalla de bienvenida: nombre del sistema, propuesta de valor en 2-3 líneas, ilustración/icono (puede ser simple)
- Pantalla de explicación del modelo Maslow simplificado: 8 áreas agrupadas en D-Needs (1-4) y B-Needs (5-8), descripción breve de cada área con ícono
- Pantalla de selección de vía: dos opciones — "Cuestionario guiado" (Stories 2.2) y "Upload de archivos psicométricos" (Story 2.3)
- Guardar en DB la vía elegida (`onboarding_method: 'questionnaire' | 'upload'`) y el estado del onboarding (`onboarding_completed: false`) en tabla `profiles` o en `users` metadata — usar Supabase `auth.users.user_metadata` para evitar crear tabla adicional
- Flujo multi-step con progress bar (3 pasos: Bienvenida → Modelo → Selección)
- Navegación: botones Siguiente/Atrás, sin poder saltar pasos
- Al seleccionar vía: redirect a Story 2.2 (`/onboarding/questionnaire`) o Story 2.3 (`/onboarding/upload`) — ambas rutas crean placeholder por ahora con mensaje "Próximamente"
- Protección del Home (`app/(app)/page.tsx`): si usuario nuevo sin diagnóstico → redirect a `/onboarding`
- Server Action `actions/onboarding.ts`: `saveOnboardingMethod(userId, method)` + `checkOnboardingStatus(userId)`
- TypeScript strict, sin `any`

**OUT of scope:**
- El cuestionario real (Story 2.2)
- El upload de archivos (Story 2.3)
- Cálculo de scores Maslow (Story 2.4)
- Vista Perfil con resultados (Story 2.5)
- Animaciones avanzadas o transiciones complejas
- Multi-idioma / i18n
- Completar el onboarding en esta story (solo flujo + selección de vía)

## Complexity

**Estimación:** S — 3 story points
- Razón: Flujo multi-step UI con Server Actions simples. La lógica de negocio es mínima (solo guardar la vía elegida). El mayor trabajo es el layout del wizard y la protección del Home. Sin dependencias externas nuevas.

## Risks

| Riesgo | Probabilidad | Impacto | Mitigación |
|--------|-------------|---------|------------|
| Redirect loop si la detección de nuevo usuario falla | Media | Alto | Usar cookie/user_metadata para marcar estado, con fallback explícito |
| Supabase `user_metadata` update falla sin permisos | Baja | Medio | Usar `supabase.auth.updateUser()` — disponible en browser client |
| Layout onboarding rompe estilos de TailAdmin | Baja | Bajo | Usar `app/(auth)/layout.tsx` existente, sin modificar `app/(app)/layout.tsx` |
| Rutas placeholder de 2.2 y 2.3 causan errores | Baja | Bajo | Crear `page.tsx` con `<p>Próximamente</p>` — suficiente para no fallar |

## Acceptance Criteria

1. Un usuario nuevo (sin `area_scores` y sin `onboarding_completed: true` en metadata) que accede a `/` es redirigido automáticamente a `/onboarding`
2. Un usuario que ya completó el onboarding (tiene `onboarding_completed: true` en user_metadata) NO es redirigido — accede directamente al Home
3. La página `/onboarding` muestra un wizard de 3 pasos con progress bar visible: paso 1 Bienvenida, paso 2 Modelo Maslow, paso 3 Selección de vía
4. El paso 1 (Bienvenida) muestra: nombre "life-os", propuesta de valor en 2-3 líneas y botón "Comenzar"
5. El paso 2 (Modelo) muestra las 8 áreas Maslow agrupadas en D-Needs (1-4) y B-Needs (5-8) con nombre e ícono de cada área
6. El paso 3 (Selección) muestra dos opciones claramente diferenciadas: "Cuestionario guiado" y "Subir archivos psicométricos", con descripción breve de cada opción
7. Al seleccionar "Cuestionario guiado" el sistema guarda `onboarding_method: 'questionnaire'` en `user_metadata` via Server Action y redirige a `/onboarding/questionnaire`
8. Al seleccionar "Subir archivos" el sistema guarda `onboarding_method: 'upload'` en `user_metadata` via Server Action y redirige a `/onboarding/upload`
9. Las rutas `/onboarding/questionnaire` y `/onboarding/upload` existen y muestran un mensaje placeholder ("En construcción — Story 2.2/2.3")
10. `npm run typecheck` PASS, `npm run lint` PASS (0 errores), `npm test` PASS — tests unitarios para `checkOnboardingStatus()` y `saveOnboardingMethod()`

## Dev Notes

### Detección de usuario nuevo

La forma más simple y sin tabla adicional es usar `supabase.auth.updateUser()` para escribir en `user_metadata`:

```typescript
// Guardar método de onboarding
await supabase.auth.updateUser({
  data: {
    onboarding_method: 'questionnaire', // o 'upload'
    onboarding_completed: false, // se pone true en Story 2.4 al terminar el diagnóstico
  }
})
```

Para leer el estado en Server Components usar `createServerClient` de `lib/supabase/server.ts`:

```typescript
const { data: { user } } = await supabase.auth.getUser()
const isNewUser = !user?.user_metadata?.onboarding_completed
```

### Protección del Home — dónde implementarla

Implementar en `app/(app)/layout.tsx` (Server Component) para que cubra todas las rutas del layout autenticado, no solo `/`:

```typescript
// app/(app)/layout.tsx
import { checkOnboardingStatus } from '@/actions/onboarding'
import { redirect } from 'next/navigation'

export default async function AppLayout({ children }) {
  const { completed } = await checkOnboardingStatus()
  if (!completed) redirect('/onboarding')
  return <>{/* sidebar + navbar + children */}</>
}
```
